# Upload / Media Domain

Status: authoritative
Last reviewed: 2026-06-01
Implementation status: implemented (Phase 3 MVP — temporary photos + private storage; audio via transcription endpoint only)

## 1. Purpose

This domain owns Houston's file and media lifecycle for operational workflows: temporary upload handling, server-side validation, linking media to the correct parent resource, authorized access, and cleanup.

It does not own Observation submission validity, AI transcription behavior, privacy policy details, or RBAC policy details. Those remain in the Observation, AI, Security / RGPD, and RBAC / Permissions domains.

## 2. MVP Scope

- Optional photos attached to Observations.
- MVP allows up to 3 photos per Observation.
- Temporary uploads before or during Observation submit.
- Server-side validation before a file is accepted or linked.
- Linking valid media to the submitted Observation.
- Cleanup of unlinked or expired uploads.
- Private-by-default media access through authorized backend flows.
- Temporary audio used only to produce editable transcription text.
- Deletion of transcription audio after transcription success or final failure.
- No image input sent to AI in MVP.

## 3. Out of Scope

- Media attachments in Establishment General Chat.
- Public media galleries, public sharing links, or publicly reachable media URLs.
- Arbitrary file manager behavior.
- Document, PDF, or video upload unless later validated.
- Long-lived audio storage.
- OCR, image understanding, or image-to-AI workflow.
- Direct browser-to-storage upload as implemented MVP behavior.
- Media editing, cropping, or library management workflow.
- Cross-resource media reuse outside validated parent-resource rules.

## 4. Core Invariants

- Files are untrusted input.
- Backend validation is authoritative for file type, size, ownership, and allowed parent context.
- Client-provided MIME type, filename, and metadata are hints only, never business authority.
- Storage keys must be generated by the backend and must not rely on raw user filenames.
- Media is private by default and remains establishment-scoped through its parent resource.
- Observation photos are optional, but a photo alone must not create a valid Observation.
- Audio is transcription-only in MVP and must not become persistent Observation media.
- Images are not sent to AI in MVP.
- Signals may surface media only through authorized linked Observation context; media must not become standalone Signal-owned content unless later validated.
- Media content must not appear in logs, notifications, realtime payloads, or durable frontend storage.
- Upload success does not mean Observation submit success.
- Orphan uploads must be cleaned up.
- Accepted image formats are candidate until implemented; target MVP formats are `jpeg`, `jpg`, `png`, and `heic` if supported by backend validation.

## 5. Main Objects

- `TemporaryUpload`
  - Unlinked file captured before the final Observation submit flow completes.
  - Exists only until it is linked, deleted, or cleaned up.

- `ObservationMedia`
  - Photo media linked to a submitted Observation.
  - Visible only through authorized parent-resource access, not as a public standalone asset.

- `TranscriptionAudio`
  - Temporary audio used only to produce editable transcription text.
  - Must not become persistent Observation media in MVP.

- `MediaMetadata`
  - Safe metadata about the file or lifecycle state.
  - Must not expose raw media content or sensitive payloads.

- `SignedMediaUrl`
  - Candidate concept for short-lived authorized access.
  - Must require backend authorization before issuance if implemented later.

- `MediaCleanupJob`
  - Candidate concept for removing orphaned, expired, or otherwise deletable media artifacts.
  - Exact job design is not validated yet.

## 6. Lifecycle / Statuses

Current code does not validate concrete upload/media status names yet. The MVP lifecycle is:

- local selection on the client
- temporary upload created
- backend validation
- validated or rejected by backend
- link to Observation on submit
- authorized access through the parent resource
- orphaned or expired candidate state
- deleted

For transcription audio, the MVP lifecycle is:

- local recording
- temporary upload or backend-controlled transport
- transcription
- validated text accepted into the Observation flow
- audio deleted

Exact persisted status fields, enums, and transition names are candidate until implemented.

## 7. Permissions

- A user may upload media only inside an authorized establishment context.
- A user may link media only to a resource they are allowed to create or update.
- Viewing media requires visibility of the authorized parent resource.
- Standalone public media access is forbidden.
- Signed access URLs, if later implemented, require backend authorization before generation.
- Active membership and establishment scoping are mandatory for any media action.
- Cross-tenant media access is forbidden.
- Broad support or admin media access is not a default product behavior and must not be assumed.

## 9. API Surface

Current API truth is `apps/api/schema.yml`.

Implemented endpoints confirmed in `apps/api/schema.yml`:

- `POST /api/v1/establishments/{establishment_id}/temporary-uploads/` — multipart photo (`jpeg`, `png`, `heic`/`heif` with server validation); private storage, no public URL in response.
- `DELETE /api/v1/establishments/{establishment_id}/temporary-uploads/{upload_id}/`
- `POST /api/v1/establishments/{establishment_id}/transcriptions/` — multipart audio only; backend OpenAI transcription; temp file deleted after each request; **not** stored as `TemporaryUpload`.

Management command: `cleanup_expired_uploads` for orphaned validated photo uploads.

Candidate API capabilities only:

- request authorized short-lived media access (read path for linked Observation media)
- direct browser-to-storage upload

## 10. Frontend Expectations

- Frontend may collect text, photos, and audio for the Observation flow, but backend validation is authoritative.
- Frontend should block obvious invalid states early, including photo-only Observation submit attempts.
- Frontend should show temporary states such as selected, uploading, failed, or linked, but must not treat those UI states as business truth.
- Frontend must not assume media is linked until backend confirmation arrives through the API.
- Frontend must not treat upload completion as Observation submit completion.
- Audio should remain local or temporary and must not be stored durably in the browser.
- Sensitive media must not be persisted in durable browser storage.
- TanStack Query owns server state for implemented APIs.
- Frontend must use generated API clients only for endpoints present in `apps/api/schema.yml`.
- Frontend must not guess media URLs or rely on direct public access patterns.

## 11. AI Agent Notes

- Inspect current upload/media code before claiming models, jobs, or endpoints exist.
- Inspect `apps/api/schema.yml` before listing any upload/media endpoint as implemented.
- Inspect the Observation, AI, Security / RGPD, and RBAC domain docs before changing submission, transcription, privacy, or access behavior.
- Do not invent signed access, direct browser-to-storage upload, cleanup jobs, or persisted status models as implemented behavior.
- Do not log media content, signed access URLs, or sensitive filenames.
- Do not send images to AI in MVP.
- Do not persist transcription audio as Observation media in MVP.
- Do not add media to Establishment General Chat in MVP.
- Do not increase the photo limit without updating product docs, backend validation, frontend validation, and tests.
- When upload/media APIs are added later, update backend authorization, OpenAPI, generated clients, tests, and this document together.
